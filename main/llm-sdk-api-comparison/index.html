    LLM Framework Public API Comparison#

    Scope: Python public APIs for OpenAI SDK, Anthropic SDK, LangChain, and LlamaIndex — covering chat/completion, streaming, async, structured outputs, tool/function calling, embeddings, and provider-abstraction patterns.

    As of: February 2026

    1. OpenAI Python SDK (openai)#

    PyPI: openai | GitHub: openai/openai-python

    Two top-level clients: OpenAI (sync) and AsyncOpenAI (async), both backed by httpx. All resources are mirrored on both clients; async methods are semantically identical but awaitable.

    1.1 Client Instantiation#

    from openai import OpenAI, AsyncOpenAI

client  = OpenAI(api_key="sk-...")       # sync
aclient = AsyncOpenAI(api_key="sk-...")  # async

    1.2 Chat Completions — client.chat.completions.create()#

    from openai.types.chat import ChatCompletion

completion: ChatCompletion = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user",   "content": "What is 2+2?"},
    ],
    temperature=0.7,
    max_tokens=1024,
    tools=[...],                              # optional tool definitions
    tool_choice="auto",                       # "none" | "auto" | specific
    response_format={"type": "json_object"},  # optional JSON mode
)

text: str = completion.choices[0].message.content

    ChatCompletion return type: 

    completion.id                           # str
completion.object                       # Literal["chat.completion"]
completion.created                      # int  (Unix timestamp)
completion.model                        # str
completion.choices                      # list[Choice]
completion.usage                        # CompletionUsage | None

# Choice fields:
choice.index                            # int
choice.finish_reason                    # "stop"|"length"|"tool_calls"|"content_filter"
choice.message.role                     # "assistant"
choice.message.content                  # str | None
choice.message.tool_calls               # list[ChatCompletionMessageToolCall] | None

    Async — identical signature, awaitable: 

    completion = await aclient.chat.completions.create(model="gpt-4o", messages=[...])

    1.3 Streaming#

    Low-level (stream=True on create()): 

    stream = client.chat.completions.create(model="gpt-4o", messages=[...], stream=True)
for chunk in stream:                    # chunk: ChatCompletionChunk
    delta = chunk.choices[0].delta.content
    if delta:
        print(delta, end="", flush=True)

    High-level context manager (preferred — accumulates final response): 

    with client.chat.completions.stream(model="gpt-4o", messages=[...]) as stream:
    for text in stream.text_stream:     # Iterator[str]
        print(text, end="", flush=True)
    final: ChatCompletion = stream.get_final_completion()

    Async high-level: 

    async with aclient.chat.completions.stream(model="gpt-4o", messages=[...]) as stream:
    async for text in stream.text_stream:   # AsyncIterator[str]
        print(text, end="", flush=True)
    final: ChatCompletion = await stream.get_final_completion()

    Low-level (stream=True) High-level (.stream())
    Sync return Iterator[ChatCompletionChunk] ChatCompletionStream (context manager)
    Async return AsyncIterator[ChatCompletionChunk] AsyncChatCompletionStream (context manager)
    Text access chunk.choices[0].delta.content stream.text_stream
    Final message manual accumulation stream.get_final_completion()

    1.4 Structured Outputs#

    Approach A — JSON mode (model constrained to emit valid JSON, schema not enforced): 

    completion = client.chat.completions.create(
    model="gpt-4o",
    messages=[...],
    response_format={"type": "json_object"},
)

    Approach B — client.beta.chat.completions.parse() (schema enforced via Pydantic): 

    from pydantic import BaseModel

class CalendarEvent(BaseModel):
    name: str
    date: str
    participants: list[str]

completion = client.beta.chat.completions.parse(
    model="gpt-4o-2024-08-06",      # structured outputs require a compatible model
    messages=[
        {"role": "system", "content": "Extract event info."},
        {"role": "user",   "content": "Alice and Bob's science fair is Friday."},
    ],
    response_format=CalendarEvent,  # Pydantic class auto-converted to JSON schema
)

event: CalendarEvent = completion.choices[0].message.parsed

    Approach B with streaming: 

    with client.beta.chat.completions.stream(
    model="gpt-4o-2024-08-06",
    messages=[...],
    response_format=CalendarEvent,
) as stream:
    final_event: CalendarEvent = stream.get_final_completion().choices[0].message.parsed

    Pydantic helper — auto-generates tool definition from a Pydantic model: 

    from openai import pydantic_function_tool

tool_def = pydantic_function_tool(CalendarEvent)  # returns dict usable in tools=[]

    1.5 Tool / Function Calling#

    tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get current weather for a city.",
            "parameters": {
                "type": "object",
                "properties": {
                    "location": {"type": "string", "description": "City name"},
                    "unit":     {"type": "string", "enum": ["celsius", "fahrenheit"]},
                },
                "required": ["location"],
            },
            "strict": True,     # optional: enforce strict schema adherence
        },
    }
]

completion = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "What's the weather in London?"}],
    tools=tools,
    tool_choice="auto",         # "none" | "auto" | {"type": "function", "function": {...}}
)

tool_call = completion.choices[0].message.tool_calls[0]
tool_call.function.name         # "get_weather"
tool_call.function.arguments    # '{"location": "London"}'  — JSON string, not dict

    1.6 Embeddings — client.embeddings.create()#

    from openai.types import CreateEmbeddingResponse

response: CreateEmbeddingResponse = client.embeddings.create(
    model="text-embedding-3-small",
    input="The food was delicious.",   # str or list[str]
    encoding_format="float",           # "float" | "base64"
    dimensions=512,                    # optional: reduce embedding dimension
)

embedding: list[float] = response.data[0].embedding
response.data[i].index                # int — position in input list
response.model                        # str
response.usage.prompt_tokens          # int
response.usage.total_tokens           # int

    1.7 Responses API (2025 — replaces Assistants)#

    Introduced March 2025, unifies Chat Completions and Assistants into a stateful endpoint:

    response = client.responses.create(
    model="gpt-4o",
    input="What is the capital of France?",     # str or list of message dicts
    tools=[{"type": "web_search_preview"}],      # built-in tools available
    previous_response_id="resp_...",             # stateful chaining
)

text: str = response.output_text       # convenience helper
response.output                        # list[OutputItem]

    Key differences from Chat Completions: - input= instead of messages= - Returns output: list[OutputItem] instead of choices - Built-in tools: web_search_preview, file_search, code_interpreter, computer_use - Stateful chaining via previous_response_id (no need to send full history)

    2. Anthropic Python SDK (anthropic)#

    PyPI: anthropic | GitHub: anthropics/anthropic-sdk-python

    Two top-level clients: Anthropic (sync) and AsyncAnthropic (async).

    2.1 Client Instantiation#

    from anthropic import Anthropic, AsyncAnthropic

client  = Anthropic(api_key="sk-ant-...")
aclient = AsyncAnthropic(api_key="sk-ant-...")

    2.2 Messages API — client.messages.create()#

    from anthropic.types import Message

message: Message = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    system="You are a helpful assistant.",    # optional top-level system param
    messages=[
        {"role": "user", "content": "What is 2+2?"}
    ],
    temperature=0.7,
    stop_sequences=["Human:"],               # optional
)

text: str = message.content[0].text

    Message return type: 

    message.id                      # str  ("msg_...")
message.type                    # "message"
message.role                    # "assistant"
message.content                 # list[ContentBlock]
message.content[0].type         # "text" | "tool_use"
message.content[0].text         # str (when type == "text")
message.model                   # str
message.stop_reason             # "end_turn" | "max_tokens" | "stop_sequence" | "tool_use"
message.usage.input_tokens      # int
message.usage.output_tokens     # int

    Key difference from OpenAI: system prompt is a top-level system= parameter (not a message in the messages list). max_tokens is required, not optional.

    Async — identical signature, awaitable: 

    message = await aclient.messages.create(model="claude-opus-4-6", max_tokens=1024, messages=[...])

    2.3 Streaming#

    Low-level (stream=True on create(), returns a context manager): 

    with client.messages.create(
    model="claude-opus-4-6", max_tokens=1024, messages=[...], stream=True
) as stream:
    for event in stream:                 # event: MessageStreamEvent
        if event.type == "content_block_delta":
            print(event.delta.text, end="", flush=True)

    High-level (preferred — client.messages.stream()): 

    with client.messages.stream(
    model="claude-opus-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Write a poem."}],
) as stream:
    for text in stream.text_stream:      # Iterator[str]
        print(text, end="", flush=True)

final: Message = stream.get_final_message()
text:  str     = stream.get_final_text()

    Async high-level: 

    async with aclient.messages.stream(
    model="claude-opus-4-6", max_tokens=1024, messages=[...]
) as stream:
    async for text in stream.text_stream:   # AsyncIterator[str]
        print(text, end="", flush=True)

final: Message = await stream.get_final_message()

    Low-level (stream=True) High-level (.stream())
    Sync return context manager → Iterator[MessageStreamEvent] MessageStream (context manager)
    Async return context manager → AsyncIterator[MessageStreamEvent] AsyncMessageStream (context manager)
    Text access event.delta.text (when content_block_delta) stream.text_stream
    Final message manual accumulation stream.get_final_message()

    2.4 Tool Use#

    Tool schema uses input_schema (JSON Schema) — different key name from OpenAI's parameters:

    tools = [
    {
        "name": "get_weather",
        "description": "Get current weather for a location.",
        "input_schema": {                   # <-- "input_schema", not "parameters"
            "type": "object",
            "properties": {
                "location": {"type": "string", "description": "City and state"},
                "unit":     {"type": "string", "enum": ["celsius", "fahrenheit"]},
            },
            "required": ["location"],
        },
    }
]

message = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    tools=tools,
    messages=[{"role": "user", "content": "What's the weather in London?"}],
)

if message.stop_reason == "tool_use":
    tool_block = message.content[0]         # ToolUseBlock
    tool_block.type                         # "tool_use"
    tool_block.name                         # "get_weather"
    tool_block.id                           # str  ("toolu_...")
    tool_block.input                        # dict  {"location": "London"}  — already parsed

# Feed result back for multi-turn:
messages.append({"role": "assistant", "content": message.content})
messages.append({
    "role": "user",
    "content": [{"type": "tool_result", "tool_use_id": tool_block.id, "content": "15°C"}],
})

    Key difference from OpenAI: tool_block.input is already a dict (not a JSON string).

    2.5 Structured Outputs (Beta, late 2025)#

    from pydantic import BaseModel

class EventInfo(BaseModel):
    name: str
    date: str
    location: str

message = client.beta.messages.parse(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Extract: Tech Conference on March 15 in NYC."}],
    response_model=EventInfo,               # Pydantic class
    betas=["structured-outputs-2025-11-13"],
)

event: EventInfo = message.parsed

    2.6 Prompt Caching (Differentiator)#

    Cache expensive content blocks using cache_control — reduces cost and latency for repeated long contexts:

    message = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "<very long system prompt or document>",
            "cache_control": {"type": "ephemeral"},  # cached for 5 min; 1h TTL opt-in
        }
    ],
    messages=[{"role": "user", "content": "Question about the above."}],
)

    3. LangChain (langchain-core)#

    PyPI: langchain-core, langchain, langchain-openai, langchain-anthropic, … | GitHub: langchain-ai/langchain

    LangChain's core value is provider abstraction via BaseChatModel — swap providers with zero downstream code changes. Every model is a Runnable.

    3.1 Core Message Types#

    from langchain_core.messages import (
    HumanMessage,
    AIMessage,
    SystemMessage,
    ToolMessage,
    BaseMessage,
    BaseMessageChunk,   # used for streaming deltas
)

    3.2 BaseChatModel — Provider-Agnostic Interface#

    All provider adapters (ChatOpenAI, ChatAnthropic, etc.) implement this interface:

    class BaseChatModel:
    def invoke(
        self,
        input: LanguageModelInput,          # str | BaseMessage | list[BaseMessage | dict]
        config: RunnableConfig | None = None,
        *,
        stop: list[str] | None = None,
        **kwargs: Any,
    ) -> BaseMessage: ...

    async def ainvoke(
        self,
        input: LanguageModelInput,
        config: RunnableConfig | None = None,
        **kwargs: Any,
    ) -> BaseMessage: ...

    def stream(
        self,
        input: LanguageModelInput,
        config: RunnableConfig | None = None,
        **kwargs: Any,
    ) -> Iterator[BaseMessageChunk]: ...

    async def astream(
        self,
        input: LanguageModelInput,
        config: RunnableConfig | None = None,
        **kwargs: Any,
    ) -> AsyncIterator[BaseMessageChunk]: ...

    def batch(
        self,
        inputs: list[LanguageModelInput],
        config: RunnableConfig | list[RunnableConfig] | None = None,
        **kwargs: Any,
    ) -> list[BaseMessage]: ...

    async def abatch(self, inputs: list[LanguageModelInput], **kwargs: Any) -> list[BaseMessage]: ...

    Usage — identical regardless of provider: 

    from langchain_openai import ChatOpenAI
from langchain_anthropic import ChatAnthropic
from langchain_core.messages import HumanMessage, SystemMessage

llm = ChatOpenAI(model="gpt-4o", temperature=0.7)
# llm = ChatAnthropic(model="claude-opus-4-6", temperature=0.7)  # drop-in replacement

response: AIMessage = llm.invoke([
    SystemMessage(content="You are a helpful assistant."),
    HumanMessage(content="What is 2+2?"),
])
text: str = response.content

# Sync streaming
for chunk in llm.stream("Write a haiku"):
    print(chunk.content, end="", flush=True)

# Async streaming
async for chunk in llm.astream("Write a haiku"):
    print(chunk.content, end="", flush=True)

    3.3 Structured Outputs — llm.with_structured_output()#

    Returns a new Runnable (not a raw call) that enforces the schema:

    from pydantic import BaseModel

class JokeResponse(BaseModel):
    setup: str
    punchline: str

structured_llm = llm.with_structured_output(
    schema=JokeResponse,            # Pydantic class | JSON schema dict | TypedDict
    method="function_calling",      # "function_calling" | "json_mode" | "json_schema"
    include_raw=False,
)

joke: JokeResponse = structured_llm.invoke("Tell me a joke.")

# With include_raw=True (debugging):
result = structured_llm.invoke("Tell me a joke.")
result["raw"]            # AIMessage  (raw LLM response)
result["parsed"]         # JokeResponse | None
result["parsing_error"]  # Exception | None

    Return type summary:

    schema include_raw Return type
    Pydantic class False instance of that class
    dict (JSON schema) False dict
    any True {"raw": AIMessage, "parsed": T\|None, "parsing_error": Exception\|None}

    3.4 Tool Calling — llm.bind_tools()#

    from langchain_core.tools import tool, StructuredTool
from pydantic import BaseModel, Field

@tool
def get_weather(location: str, unit: str = "celsius") -> str:
    """Get the current weather for a location."""
    return f"15 degrees {unit} in {location}"

llm_with_tools = llm.bind_tools([get_weather])        # any mix of tools accepted
response: AIMessage = llm_with_tools.invoke("Weather in Paris?")
response.tool_calls   # list[dict]: [{"name": ..., "args": {...}, "id": "..."}]

    bind_tools() accepts: @tool-decorated functions, StructuredTool, Pydantic models, plain functions, or raw JSON schema dicts — all interchangeably.

    class SearchInput(BaseModel):
    query: str = Field(description="Search query")
    max_results: int = Field(default=5)

search_tool = StructuredTool.from_function(
    func=lambda query, max_results=5: f"Results for '{query}'",
    name="web_search",
    description="Search the internet",
    args_schema=SearchInput,
)

    3.5 Embeddings — Embeddings Abstract Base#

    from langchain_core.embeddings import Embeddings

class Embeddings:
    def embed_documents(self, texts: list[str]) -> list[list[float]]: ...
    def embed_query(self, text: str) -> list[float]: ...
    async def aembed_documents(self, texts: list[str]) -> list[list[float]]: ...
    async def aembed_query(self, text: str) -> list[float]: ...

    from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

query_vec: list[float]        = embeddings.embed_query("What is LangChain?")
doc_vecs:  list[list[float]]  = embeddings.embed_documents(["doc one", "doc two"])

    Swap OpenAIEmbeddings → HuggingFaceEmbeddings, CohereEmbeddings, etc. with zero downstream changes.

    3.6 LCEL — Runnable Composition with |#

    Every LangChain component implements Runnable, enabling pipe-based chain composition:

    from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    ("human", "{input}"),
])

chain = prompt | ChatOpenAI(model="gpt-4o") | StrOutputParser()

result: str = chain.invoke({"input": "Tell me about LangChain."})

# Streaming propagates through the entire chain:
for chunk in chain.stream({"input": "Tell me about LangChain."}):
    print(chunk, end="", flush=True)

    4. LlamaIndex (llama-index-core)#

    PyPI: llama-index-core + provider packages (llama-index-llms-openai, llama-index-llms-anthropic, …) | GitHub: run-llama/llama_index

    LlamaIndex explicitly separates text completion from chat, and has structured_predict() as a first-class method.

    4.1 Core Data Types#

    from llama_index.core.base.llms.types import (
    ChatMessage,
    ChatResponse,
    CompletionResponse,
    MessageRole,
    CompletionResponseGen,      # Generator[CompletionResponse, None, None]
    ChatResponseGen,            # Generator[ChatResponse, None, None]
    CompletionResponseAsyncGen, # AsyncGenerator[CompletionResponse, None]
    ChatResponseAsyncGen,       # AsyncGenerator[ChatResponse, None]
)

    ChatMessage: 

    ChatMessage(
    role=MessageRole.USER,        # USER | ASSISTANT | SYSTEM | TOOL | FUNCTION | CHATBOT
    content="What is 2+2?",
    additional_kwargs={},         # tool_calls, etc.
)

    ChatResponse: 

    response.message                  # ChatMessage (role=ASSISTANT)
response.message.content          # str
response.raw                      # dict | None  (raw provider JSON)
response.delta                    # str | None   (non-None only during streaming)
response.additional_kwargs        # dict

    CompletionResponse: 

    response.text                     # str
response.raw                      # dict | None
response.delta                    # str | None   (non-None only during streaming)

    4.2 LLM Interface — Core Methods#

    LlamaIndex explicitly separates complete() (text in → text out) from chat() (messages in → message out):

    Sync: 

    # Text completion (string prompt)
response: CompletionResponse = llm.complete("The capital of France is")

# Chat (list of ChatMessage)
response: ChatResponse = llm.chat([
    ChatMessage(role=MessageRole.SYSTEM, content="You are helpful."),
    ChatMessage(role=MessageRole.USER,   content="What is 2+2?"),
])

# Streaming completion
gen: CompletionResponseGen = llm.stream_complete("Once upon a time")
for partial in gen:               # each: CompletionResponse with .delta set
    print(partial.delta, end="")

# Streaming chat
gen: ChatResponseGen = llm.stream_chat(messages)
for partial in gen:               # each: ChatResponse with .delta set
    print(partial.delta, end="")

    Async: 

    response: CompletionResponse = await llm.acomplete("The capital of France is")
response: ChatResponse        = await llm.achat(messages)

async for partial in await llm.astream_complete("Once upon a time"):
    print(partial.delta, end="")

async for partial in await llm.astream_chat(messages):
    print(partial.delta, end="")

    4.3 Structured Prediction — first-class structured_predict()#

    structured_predict() is a core method on every LLM, not a wrapper:

    from pydantic import BaseModel
from llama_index.core import PromptTemplate

class Album(BaseModel):
    name: str
    artist: str
    year: int

prompt_tmpl = PromptTemplate("Generate a famous album by {artist}")

# Sync — returns fully validated instance
album: Album = llm.structured_predict(
    output_cls=Album,
    prompt=prompt_tmpl,
    artist="The Beatles",
)

# Async
album: Album = await llm.astructured_predict(
    output_cls=Album,
    prompt=prompt_tmpl,
    artist="The Beatles",
)

# Streaming — yields partially complete instances as fields fill in
gen = llm.stream_structured_predict(output_cls=Album, prompt=prompt_tmpl, artist="Radiohead")
for partial_album in gen:
    print(partial_album)            # incomplete Album instance; grows with each chunk

# Async streaming
async for partial_album in await llm.astream_structured_predict(
    output_cls=Album, prompt=prompt_tmpl, artist="Radiohead"
):
    print(partial_album)

    4.4 Tool / Function Calling#

    from llama_index.core.tools import FunctionTool

def add(a: int, b: int) -> int:
    """Add two numbers together."""
    return a + b

async def async_add(a: int, b: int) -> int:
    return a + b

add_tool = FunctionTool.from_defaults(
    fn=add,
    async_fn=async_add,           # optional async variant
    name="add",                   # defaults to function name
    description="Add two integers",
)

# invoke_callable: LLM picks the right tool, calls it, returns result
response = llm.invoke_callable(
    tools=[add_tool],
    user_msg="What is 7 + 15?",
    verbose=True,
)

    ToolSpec — groups related tools into a reusable spec: 

    from llama_index.core.tools import BaseToolSpec

class MyAPIToolSpec(BaseToolSpec):
    spec_functions = ["search", "get_details"]

    def search(self, query: str) -> str:
        """Search the API."""
        ...

    def get_details(self, id: str) -> dict:
        """Get details for an item."""
        ...

tools = MyAPIToolSpec().to_tool_list()

    4.5 Embeddings — BaseEmbedding#

    LlamaIndex handles batching internally via embed_batch_size:

    class BaseEmbedding:
    embed_batch_size: int = 10         # auto-batches large inputs

    4.6 Custom LLM — CustomLLM#

    Minimum required: complete(), stream_complete(), and metadata property:

    from llama_index.core.llms import CustomLLM, LLMMetadata
from llama_index.core.base.llms.types import CompletionResponse, CompletionResponseGen

class MyLLM(CustomLLM):
    context_window: int = 4096
    num_output: int = 256
    model_name: str = "my-model"

    @property
    def metadata(self) -> LLMMetadata:
        return LLMMetadata(
            context_window=self.context_window,
            num_output=self.num_output,
            model_name=self.model_name,
        )

    def complete(self, prompt: str, **kwargs: Any) -> CompletionResponse:
        return CompletionResponse(text=my_model_api(prompt))

    def stream_complete(self, prompt: str, **kwargs: Any) -> CompletionResponseGen:
        for chunk in my_model_api_stream(prompt):
            yield CompletionResponse(text=chunk, delta=chunk)

    Key Architectural Observations#

    OpenAI SDK is a thin, direct REST wrapper. The sync/async split is entirely at the client level (OpenAI vs AsyncOpenAI). The 2025 Responses API (client.responses.create) supersedes Assistants for stateful multi-turn workflows; Chat Completions remains the stable path for standard use. client.beta.chat.completions.parse() auto-handles Pydantic→JSON schema conversion and populates a .parsed field on the response. Tool arguments arrive as a JSON string requiring manual json.loads().

    Anthropic SDK follows the same sync/async client split. The critical API difference: tool definitions use input_schema (not parameters), and tool_block.input arrives already as a parsed dict (no json.loads() needed). System prompt is a top-level system= parameter, not a message in the messages array. max_tokens is required. Prompt caching via cache_control blocks is the key cost-reduction differentiator.

    LangChain every model is a Runnable implementing invoke / ainvoke / stream / astream / batch / abatch. The abstraction is the primary value: swap ChatOpenAI for ChatAnthropic with zero downstream changes. with_structured_output() returns a new composable Runnable, not a raw response. bind_tools() accepts heterogeneous tool formats (functions, Pydantic, dicts). LCEL's | operator composes prompt → LLM → parser into streaming-capable chains.

    LlamaIndex explicitly separates string-in/string-out complete() from messages-in/message-out chat() (unlike LangChain which only exposes invoke). structured_predict() is a first-class core method on every LLM, not a wrapper. CustomLLM only requires complete() + stream_complete() + metadata. The streaming delta is exposed as .delta on every CompletionResponse / ChatResponse chunk. BaseEmbedding handles batching automatically via embed_batch_size.
